## 版本概要

v0.6 是 audio.cpp-runtime-go 的第一個正式版本，涵蓋從專案初始化到真實 ASR 端到端驗證的完整六階段開發。Runtime 提供 OpenAI-compatible HTTP API、Job 排程系統、Model Registry、SQLite 持久化儲存、跨平台 Process 管理，並通過真實 Citrinet ASR 模型的 Smoke Test 驗證。

---

## 新增功能

### Phase 1 — 專案初始化與基本架構（Commit `ca1a3e9`）

- **專案 Scaffold**: Go module (`github.com/user/audiocppruntime`)、`.gitignore`、`build.bat`
- **基本 Runtime 管理**: `internal/runtime/manager.go` — Runtime 生命週期管理、`internal/runtime/config.go` — 基本配置
- **內建 Web Server**: `internal/server/` — 基於 gorilla/mux 的 HTTP server，含 HTML template 引擎、檔案上傳處理
- **Model Catalog**: `internal/catalog/` — 靜態 JSON 模型目錄系統
- **Submodule**: `audio.cpp`、`audio.cpp-webui` 作為 Git submodule 引用
- **依賴**: `github.com/gorilla/mux v1.8.1`、Go 1.22

### Phase 2 — API Server 與 Job System（Commit `7e2330c`）

- **OpenAI-compatible HTTP API** (`internal/api/`):
  - `POST /v1/audio/speech` — TTS（Text-to-Speech，OpenAI 相容格式）
  - `POST /v1/audio/transcriptions` — ASR（Automatic Speech Recognition，OpenAI 相容格式）
  - `POST /v1/tts`、`POST /v1/asr` — 內部 API（支援 JSON 與 multipart/form-data）
  - `GET /v1/health` — 健康檢查（含 runtime 狀態、child PID、models count）
  - `GET /v1/models`、`GET /v1/models/{id}` — 模型查詢
  - `GET /v1/voices` — 語音查詢
  - `GET /v1/capabilities` — Backend 能力查詢
  - `POST /v1/tasks/run`、`POST /v1/tasks/stream` — 通用任務執行
  - `POST /v1/shutdown` — Runtime 優雅關閉
- **Job System** (`internal/jobs/`):
  - **Job Model**: 優先級、狀態追蹤（pending → queued → running → completed/failed）
  - **Priority Queue**: 基於 heap 的優先級佇列
  - **Job Manager**: CRUD 操作、持久化、列表查詢
  - **Worker Pool**: 多 Worker 並行處理，支援優雅關閉
- **audio.cpp Client** (`internal/audiocpp/`):
  - **Client**: HTTP client 封裝 audio.cpp server 的所有端點（Speech、Transcription、Health、Models）
  - **CLI Executor**: CLI 命令執行器，作為 HTTP API 的 fallback
  - **Capabilities**: 靜態能力映射（task → capabilities、capabilities → task）
  - **Health**: Child process 健康狀態查詢
  - **錯誤處理**: 統一錯誤類型與 HTTP error mapping
- **跨平台 Process 管理** (`internal/platform/`):
  - **Windows**: `CreateProcess`、`KillProcessTree`（`taskkill /T /F`）、`IsProcessRunning`、`GetProcessPath`
  - **Unix**: `SIGTERM`/`SIGKILL` 信號發送、Process Group 管理
- **YAML 配置系統** (`internal/config/`):
  - `Config` 結構體：Server / AudioCpp / Storage / Models / Outputs / Jobs 區段
  - `AudioCppConfig`：server_path、cli_path、working_dir、backend、device、timeout、auto_restart
  - 路徑解析：`ResolvePaths(baseDir)` 自動相對路徑解析
  - 配置驗證：server_path 存在性檢查、port 範圍、timeout 合理性
  - 完整單元測試覆蓋（`internal/config/config_test.go`）

### Phase 3 — Model Registry 與 Storage（Commit `7e2330c`）

- **Model Registry** (`internal/models/`):

## 破壞性修改

### API 變更

| 變更 | 說明 | 遷移方式 |
|------|------|----------|
| `POST /v1/tts` 保留為內部 API | OpenAI-compatible `POST /v1/audio/speech` 為主要端點 | 新開發請使用 `/v1/audio/speech` |
| `POST /v1/asr` 保留為內部 API | OpenAI-compatible `POST /v1/audio/transcriptions` 為主要端點 | 新開發請使用 `/v1/audio/transcriptions` |
| `GET /v1/health` 回傳格式擴充 | 新增 `audiocpp_pid`、`audiocpp_state`、`audiocpp_alive` | 向後相容，舊欄位不變 |
| `POST /v1/shutdown` 回傳格式變更 | 從單一 boolean 改為 `ShutdownResult` 三欄位結構 | 客戶端需更新 shutdown 回應解析 |

### Config 變更

| 變更 | 說明 | 遷移方式 |
|------|------|----------|
| `audiocpp.lazy_load` 新增 | 預設 `false` | 舊配置不受影響 |
| `audiocpp.model_spec_override` 新增 | 可選，預設空 | 舊配置不受影響 |
| `audiocpp.models` 新增 | 可選多模型配置 | 舊配置不受影響 |
| SQLite driver 變更 | `modernc.org/sqlite` → `github.com/glebarez/go-sqlite` | 無需手動遷移，資料庫格式相容 |
| Go 版本最低要求 | Go 1.25+（原 Go 1.22） | 需更新開發環境至 Go 1.25+ |

### 行為變更

| 變更 | 說明 | 遷移方式 |
|------|------|----------|
| Shutdown 語義 | API 成功 ≠ graceful shutdown 成功，需檢查 `GracefulExited` 欄位 | 更新監控腳本 |
| Smoke 判定標準 | 加入內容匹配門檻（≥5 words / ≥50%）+ HTTP Status 檢查 | 既有 smoke 結果可能從 PASS 變為 FAIL |
| `real_smoke_test.ps1` | 完全取代 `smoke_test.bat`，路徑與參數不同 | 使用 `scripts/real_smoke_test.ps1` |
| 配置相對路徑解析 | 所有路徑基於 `config.yaml` 所在目錄解析 | 若使用絕對路徑則不受影響 |

## 已知限制（Known Gaps）

詳細技術債清單請參閱 [`docs/TECHNICAL_DEBT.md`](../TECHNICAL_DEBT.md)。以下為 v0.6 主要已知限制：

| 編號 | 名稱 | 優先級 | 影響 | 建議 Phase |
|------|------|--------|------|-----------|
| TD-005 | 僅支援 audio.cpp 單一 Backend（無抽象層） | **P0** | 無法抽換其他推論引擎 | Phase 7 |
| TD-002 | audio.cpp child graceful shutdown 未實作 | **P1** | Smoke 驗收無法完全優雅關閉 | Phase 7 |
| TD-006 | GPU Backend 未實作（僅 CPU） | **P1** | 大型模型推論速度慢 | Phase 8 |
| TD-007 | Streaming Pipeline 未實作 | **P1** | 無法即時 ASR/TTS | Phase 7 |
| TD-012 | HTTPS / API 認證缺失 | **P1** | 明文傳輸，無存取控制 | Phase 7 |
| TD-001 | Backend Version Endpoint 缺失 | P2 | Contract test 無法驗證版本 | Phase 7 |
| TD-003 | Race Test 無法在 Windows 執行 | P2 | 無法偵測 data race | Phase 7 |
| TD-004 | Backend Capabilities 缺乏可插拔擴充 | P2 | 新增能力需修改 Go 原始碼 | Phase 7 |
| TD-008 | Batch Inference 未支援 | P2 | 高吞吐場景 GPU 利用率低 | Phase 8 |
| TD-009 | Hot Reload 未實作 | P2 | 模型更新需重啟服務 | Phase 8 |
| TD-010 | Model Cache 未實作 | P2 | 大型模型載入時間長 | Phase 8 |
| TD-011 | `lazy_load=true` 時 `model_spec_override` 失效 | P2 | 使用情境受限 | Phase 7 |
| TD-013 | 日誌系統缺乏結構化 | P2 | 生產除錯困難 | Phase 7 |
| TD-014 | 測試覆蓋率與 CI 缺口 | P2 | 回歸風險 | Phase 7 |
| TD-015 | 缺乏持續監控 Metrics | P2 | 無即時健康趨勢 | Phase 7 |

**總計**: 15 項技術債（P0×1、P1×4、P2×10），預估總工作量約 18 人/週。

### 特殊記錄

- **Version Contract**: `TestBackendContract_Version` 為唯一 `t.Skip`，原因為 audio.cpp 上游未實作 `/v1/version` endpoint。不計入 Contract Test PASS 計數。
- **Race Test**: `go test -race ./...` 在 Windows 環境因缺少 gcc/cgo toolchain 無法執行。記錄為 NOT RUN，不影響驗收。
- **Top-level `/tests` 套件**: 因既有測試問題，仍排除在 CI 之外，需在 Phase 7 修復後納入。

## 升級指南

### 從 v0.5（Phase 1~5）升級至 v0.6

v0.6 是從初始版本（Phase 1）到 Phase 6 封板的完整版本，沒有「從 v0.5 升級」的路徑。若您從 Phase 1~5 的程式碼基礎開始，請直接 checkout 最終 commit `f009937`（或 `master`）。

### 首次部署

1. **環境需求**:
   - Go 1.25+（建議 go1.26.4）
   - Windows（完整支援）或 Linux/macOS（有限支援）
   - Git（用於 submodule）

2. **取得程式碼**:
   ```bash
   git clone --recursive https://github.com/liuxb99/audiocpp-runtime-go
   cd audiocpp-runtime-go
   ```

3. **建置**:
   ```bash
   .\build.bat
   # 或個別建置
   go build -o bin\audiocpp-runtime.exe .\cmd\audiocpp-runtime\
   go build -o bin\audiocppctl.exe .\cmd\audiocppctl\
   ```

4. **配置**:
   - 複製 `configs/config.example.yaml` 為 `configs/config.yaml`
   - 修改 `audiocpp.server_path` 指向真實 `audiocpp_server.exe` 路徑
   - 若有真實模型，設定 `audiocpp.models` 區段

5. **執行**:
   ```bash
   .\scripts\run.bat
   # 或直接執行
   .\bin\audiocpp-runtime.exe -config .\configs\config.yaml
   ```

6. **驗證**:
   ```bash
   curl http://127.0.0.1:8091/v1/health
   ```

### 真實 ASR 驗證（含 Citrinet 模型）

若需執行真實 ASR Smoke Test：

1. 依照 [`docs/REAL_AUDIOCPP_BUILD.md`](../REAL_AUDIOCPP_BUILD.md) 建置含 ASR 支援的 `audiocpp_server.exe`
2. 依照 [`docs/REAL_MODEL_CITRINET.md`](../REAL_MODEL_CITRINET.md) 下載 Citrinet 模型
3. 記錄模型 SHA256 至 `docs/model-sha256.json`
4. 執行：
   ```powershell
   .\scripts\real_smoke_test.ps1
   ```
5. 檢查 `artifacts/smoke/result.md` 中的 `REAL_SMOKE_PASS`

### 配置注意事項

- **路徑**: 建議使用相對於 `config.yaml` 的相對路徑，`ResolvePaths()` 會自動解析
- **lazy_load**: 需要自訂模型規格時請設為 `false`（已知限制 TD-011）
- **auto_restart**: 建議啟用（預設 `true`），Runtime 會在 child process 崩潰時自動重啟
- **workers**: 預設 `1`，可依硬體效能調整，但需注意 audio.cpp 的併發限制

### 已知不相容

- **CGO 依賴**: 若先前使用 `modernc.org/sqlite` 的直接 CGO 模式，遷移至 `github.com/glebarez/go-sqlite` 後無需修改程式碼，資料庫格式完全相容
- **舊版 Smoke Test**: `smoke_test.bat` 已被 `scripts/real_smoke_test.ps1` 取代，舊腳本不再維護
- **舊版 Server**（`internal/server/`）: 已在 Phase 2 完全移除，無法使用舊版 `-port` / `-host` / `-bundle` 等旗標
