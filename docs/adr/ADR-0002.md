# ADR-0002: Backend Contract

## 狀態

**Accepted** — 實作於 Phase 6D/6E，提交 `e4dbfb2` 和 `074d4e7`。

## Context

`audio-cpp-runtime-go` 的核心職責是管理 `audio.cpp` 子程序的完整生命週期。專案初期，`internal/audiocpp/process.go` 的 `Process` 結構體直接承擔了啟動、停止、健康檢查等所有職責，但存在以下問題：

1. **無統一介面**：Runtime 直接依賴具體的 `*audiocpp.Process`、`*audiocpp.Client`、`*audiocpp.CLIExecutor` 型別，沒有定義 Backend 抽象介面（Interface）。
2. **行為無契約**：沒有文件規範「一個 Backend 必須支援哪些操作」。`Stop()` 的行為、`Pid()` 的返回值語義等全靠實作決定。
3. **無法抽換**：若要支援 Whisper.cpp、TensorRT 或其他推論引擎，需大幅修改 Runtime 與 API 層——這被記錄為 TD-005（P0 優先級）。
4. **版本缺口**：audio.cpp server 未實作 `/v1/version` 端點，`TestBackendContract_Version` 被迫以 `t.Skip()` 跳過，記錄為 TD-001。

Phase 6D 的需求要求建立 Backend Contract Test，確保任何 Backend 實作都遵守同一組行為約定。

## Decision

採用 **Contract Test 驅動的 Backend 介面定義** 策略——不先定義 Go Interface，而是先用 Contract Test 固定行為，再從測試反推介面。

### Backend Contract Test 項目

定義在 `internal/audiocpp/backend_test.go` 中，每個測試對應一個 contract 項目：

| Contract | 測試方法 | 目前狀態 |
|----------|---------|---------|
| **Start** | `TestBackendContract_Start` | ✅ PASS — 啟動後狀態為 `Starting` |
| **Ready** | `TestBackendContract_Ready` | ✅ PASS — WaitForReady 後狀態為 `Running` |
| **Health** | `TestBackendContract_Health` | ✅ PASS — HTTP 200，status=="ok" |
| **Stop** | `TestBackendContract_Stop` | ✅ PASS — 停止後狀態為 `Stopped` |
| **PID** | `TestBackendContract_PID` | ✅ PASS — PID > 0 |
| **Version** | `TestBackendContract_Version` | ⚠️ SKIP — Known Gap，audio.cpp 未實作 `/v1/version` |
| **Capabilities** | `TestBackendContract_Capabilities` | ✅ PASS — 驗證 task→capability 對映完整 |
| **ForceStop** | `TestBackendContract_ForceStop` | ✅ PASS — 強制終止後 child 退出、狀態 Stopped、冪等 |

### 對應的 Process 方法

`internal/audiocpp/process.go` 實作了 contract 所需的所有方法：

```go
func (p *Process) Start(ctx context.Context) error
func (p *Process) WaitForReady(ctx context.Context, timeout time.Duration) error
func (p *Process) Stop() error
func (p *Process) StopGraceful() bool
func (p *Process) ForceStop() error
func (p *Process) Pid() int
func (p *Process) State() ProcessState
```

### Shutdown Contract

除了 Backend Contract，另有獨立的 Shutdown Contract（定義於 `internal/runtime/contract_test.go`），規範 Runtime 層級的關閉步驟：

```
RequestAccepted → StopWorkers → FlushQueue → StopChild → SaveState → CloseDB → StopHTTP → ExitMain
```

每個步驟記錄 timeout、error、duration、result。

### Version Known Gap 處理

`TestBackendContract_Version` 明確標記為 `t.Skip("KNOWN GAP: backend version endpoint not implemented")`，此為 audio.cpp 上游程式碼的缺口，非 Go Runtime 本身的問題。在最終報告中獨立列出此項，不計入 PASS 計數。

## Alternatives

### 無介面直接呼叫

Runtime 直接使用 `*audiocpp.Process` 的方法，不定義任何介面或 contract。

- **優點**：實作簡單，無抽象層開銷。
- **缺點**：無法抽換 Backend；新 Backend 開發者無行為指引；無自動化驗證確保行為一致。
- **棄用原因**：TD-005 已將「單一 Backend 限制」標記為 P0 緊急，無法繼續忽略。

### Plugin 架構

定義完整的 Backend Interface（Go interface），以 plugin 方式動態載入。

- **優點**：完全解耦，可熱插拔。
- **缺點**：需設計 Factory、動態載入機制、版本相容性檢查；Go plugin 在 Windows 上支援有限（`plugin` package 僅支援 macOS/Linux）。Phase 6D/6E 的時間約束不允許引入此複雜度。
- **棄用原因**：過早抽象。先以 Contract Test 固定行為，待 Phase 7 再正式提取 Interface。

### 僅 Runtime 層 Contract（無 Process 層）

只在 Runtime 層測試完整生命週期，不單獨測試 Process。

- **缺點**：無法獨立驗證 Process 的行為；Process 的 `ForceStop`、`StopGraceful`、`WaitForReady` 等細節被 Runtime 包裝，測試覆蓋不足。
- **棄用原因**：Process 層包含了子程序管理的核心邏輯（generation、supervise、auto-restart），需要獨立的 contract 測試。

## Consequences

### 正面影響

1. **行為一致性**：任何 Backend 實作通過 Contract Test 即可保證符合 Runtime 期望。
2. **測試即文件**：Contract Test 本身就是 Backend 的行為規範，比文件更精確。
3. **Known Gap 可追溯**：Version 的 SKIP 狀態有明確原因、影響、優先級，不會被遺忘。
4. **Shutdown 可觀測**：每個 shutdown 步驟都有記錄，可事後分析哪一步出了問題。

### 負面影響

1. **缺少正式 Interface**：目前仍是隱式契約（Duck Typing），尚未提取為 Go Interface。Phase 7 需要在 Contract Test 的基礎上完成 Interface 萃取。
2. **Version 為永久缺口**：除非 audio.cpp 上游新增版本端點，否則無法完全關閉此 contract。這是一個外部依賴的缺口。
3. **測試維護成本**：Contract Test 需要 fake child process，測試環境設定較複雜（`FAKE_AUDIOCPP_CHILD=1`）。未來新增 Backend 時需確保 fake child 也能模擬該 Backend 的行為。

### Trade-off

Contract Test 先行、Interface 後補的策略，在 Phase 6 的時間約束下取得了正確的平衡：先確保行為正確且可驗證，再在 Phase 7 進行正式抽象。TD-005 將在 Phase 7 被解決。
