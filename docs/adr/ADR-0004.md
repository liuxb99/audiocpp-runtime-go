# ADR-0004: State Machine

## 狀態

**Accepted** — 實作於 Phase 6D，提交 `e4dbfb2`。

## Context

Phase 6D 之前，Runtime 的生命週期管理存在以下問題：

1. **無集中狀態管理**：`Runtime` 結構體雖然有 `state` 欄位，但沒有統一的遷移控制。各個方法直接修改 `r.state`，沒有任何校驗。
2. **非法遷移可能發生**：例如從 `StateCreated` 可以直接跳到 `StateRunning`，或 `StateRunning` 可以再次執行 `StartAudioCpp()`。
3. **重複進入可能**：`Shutdown()` 可以在已停止的 Runtime 上再次執行，導致 panic 或 undefined behavior。
4. **Phase 6C 的 shutdown 語義修正** 暴露了狀態管理的不足——需要區分「從哪個狀態進入 shutdown」以及「shutdown 是否已經完成」。

Phase 6D 的需求第一條即為：「整理成真正的狀態機：Created → Initializing → Starting audio.cpp → Ready → Running → Stopping → Stopped。所有狀態遷移必須：單一路徑、禁止跳躍、禁止重複進入。」

## Decision

採用 **單一路徑狀態機** 作為 Runtime 狀態管理模型，嚴格控制所有狀態遷移。

### 核心設計

定義於 `internal/runtime/state.go`：

```go
type RuntimeState int32

const (
    StateCreated      RuntimeState = 0
    StateInitializing RuntimeState = 1
    StateStarting     RuntimeState = 2
    StateReady        RuntimeState = 3
    StateRunning      RuntimeState = 4
    StateStopping     RuntimeState = 5
    StateStopped      RuntimeState = 6
)

var validTransitions = map[RuntimeState][]RuntimeState{
    StateCreated:      {StateInitializing, StateStopping},
    StateInitializing: {StateStarting, StateStopping},
    StateStarting:     {StateReady, StateStopping},
    StateReady:        {StateRunning, StateStopping},
    StateRunning:      {StateStopping},
    StateStopping:     {StateStopped},
    StateStopped:      {},
}
```

### 三項核心原則

1. **單一路徑**：正常生命週期只有一個前進方向 `Created → Initializing → Starting → Ready → Running → Stopping → Stopped`，不能跳過任何狀態。
2. **禁止跳躍**：不能從 `Created` 直接到 `Running`，不能從 `Starting` 直接到 `Stopped`（除了 shutdown 捷徑）。
3. **禁止重複進入**：`validTransitions` 中沒有從某狀態回到自身的路徑（無 self-loop），因此無法重複初始化、重複啟動。

### Shutdown 捷徑

從任何活躍狀態都可直接轉入 `Stopping`：

```go
func (r *Runtime) transitionFromAnyToStoppingLocked() error {
    allowedSources := []RuntimeState{
        StateCreated, StateInitializing, StateStarting, StateReady, StateRunning,
    }
    for _, src := range allowedSources {
        if r.state == src {
            r.state = StateStopping
            return nil
        }
    }
    return fmt.Errorf("cannot transition from %s to stopping", StateString(r.state))
}
```

這確保 shutdown 請求永遠不會被拒絕，同時保持「Stopping → Stopped」為單一路徑。

### 雙重檢查機制

- `transition(from, to RuntimeState)`：取得鎖 → 驗證當前狀態 == from → 查表驗證 to 合法 → 修改狀態。
- `transitionLocked(from, to RuntimeState)`：同上但假設鎖已被呼叫者持有，用於 shutdown 等需要原子操作多個步驟的場景。

### 狀態查詢

```go
func (r *Runtime) CurrentState() RuntimeState  // 公開方法
func (r *Runtime) currentState() RuntimeState  // 內部方法（私有）
```

所有外部查詢統一透過 `CurrentState()`，不再允許各元件自行判斷狀態。

### 測試覆蓋

狀態機的 contract test 包括：

| 測試 | 驗證內容 |
|------|---------|
| `TestRuntimeLifecycle_FullCycle` | 完整路徑：Created→Initializing→Starting→Ready→Running→Stopped |
| `TestStateTransition_DoubleShutdown` | 兩次 shutdown 不 panic，最終狀態為 Stopped |
| `TestStateTransition_ShutdownBeforeStart` | 從 Created 直接 shutdown 也正常 |
| `TestStateTransition_InvalidTransition` | 在 Initializing 嘗試 StartWorkers（預期等待或忽略） |
| `TestBackend_StartupFailure` | 啟動失敗不留下畸形狀態 |
| `TestBackend_ChildCrash` | child crash 不影響 Runtime 狀態機正確性 |

## Alternatives

### Event-driven 架構

透過事件匯流排（Event Bus）廣播狀態變更，各元件回應事件。

- **優點**：元件間鬆耦合；可動態加入新的 event handler。
- **缺點**：狀態變更的時序難以預測；事件回呼可能形成隱式的狀態機；debug 時需要追蹤事件流而非簡單的函式呼叫堆疊。
- **棄用原因**：Phase 6D 的需求明確要求「單一路徑、禁止跳躍、禁止重複進入」，事件驅動無法提供同樣強度的保證。

### Lock-free 模型

以 atomic 操作和 CAS（Compare-And-Swap）實現無鎖狀態機。

- **優點**：高效能，無鎖競爭。
- **缺點**：`transitionFromAnyToStoppingLocked()` 等複雜邏輯在 CAS 模型中難以實作；需要 spin lock 或 retry loop；對 Go 的 memory model 要求高。
- **棄用原因**：Runtime 狀態機的呼叫頻率極低（整個生命週期約 7 次遷移），鎖競爭開銷可忽略。無鎖帶來的好處遠低於其複雜度成本。

### 鬆散狀態管理（維持原狀）

不引入狀態機，各方法自由修改 `r.state`，僅靠程式碼審查保證正確。

- **缺點**：Phase 6C 已證明此路線不可靠——double shutdown、shutdown-before-start、非法狀態跳躍等問題皆源於缺乏強制機制。
- **棄用原因**：直接違背 Phase 6D 的核心需求。

## Consequences

### 正面影響

1. **可預測的行為**：任何程式路徑都能明確知道當前處於哪個狀態、下一步可以到哪個狀態。
2. **強制安全**：編譯器無法檢查但 runtime 的 `transition()` 函式保證了非法遷移會被攔截並返回 error。
3. **可測試性**：狀態機的每個轉移都可以獨立測試，邊界情況（double shutdown、shutdown before start）都有對應的 contract test。
4. **穩定性提升**：Phase 6D 之後未再發生因狀態管理不當導致的 crash 或 panic。

### 負面影響

1. **轉移延遲略有增加**：每次遷移需取得 mutex lock，但實測延遲在微秒級，不影響整體效能。
2. **新增狀態的成本**：若要新增狀態（如 `Paused`、`Degraded` 模式），需修改 `validTransitions`、`StateString`、`transitionFromAnyToStoppingLocked` 以及所有相關測試。
3. **與 Process 狀態機的重疊**：Runtime 有自己的狀態機，`Process` 也有自己的 `ProcessState`。兩者的關係需透過 `supervise()` goroutine 維持一致性。例如 `Process` 可能因 crash 進入 `StateCrashed`，但 Runtime 仍然處於 `StateRunning`，直到 supervisor 觸發重啟或 shutdown。

### Trade-off

嚴格的狀態機約束了程式結構的靈活性，但換來了生命週期管理的確定性。對於一個需要管理外部子程序、處理非同步 crash 與重啟的系統，確定性比靈活性更重要。Phase 6D 的實作經驗證實：狀態機是 Runtime 穩定性的基石，而非束縛。
