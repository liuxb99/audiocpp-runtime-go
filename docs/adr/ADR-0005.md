# ADR-0005: Diagnostics API

## 狀態

**Accepted** — 實作於 Phase 6D，提交 `e4dbfb2`。

## Context

Phase 6D 之前，Runtime 內部狀態的可觀測性嚴重不足：

1. **無統一觀測點**：想知道 Runtime 的當前狀態、worker 數量、佇列長度等資訊，需要分別查詢不同元件。
2. **僅 `/v1/health`**：Phase 6B 曾在 `/v1/health` 中加入 `audiocpp_pid`、`audiocpp_state`、`audiocpp_alive` 等欄位，但 health endpoint 的職責是「服務是否活著」，不適合承載完整的內部狀態。
3. **Shutdown 過程不透明**：Phase 6C 修正 shutdown 語義後，雖然 `ShutdownSchedule` 記錄了每一步的結果，但沒有 API 可以查詢 shutdown 過程。
4. **無法紀錄快照**：沒有機制將某個時刻的 Runtime 狀態序列化為檔案，供事後分析。

Phase 6D 的需求明確要求增加 Runtime 可觀測性，同時強調「不要加入 Prometheus，保持輕量」。

## Decision

採用 **輕量 Diagnostics API** 作為 Runtime 可觀測性方案。

### 設計要點

1. **單一端點**：`GET /v1/runtime/diagnostics` 回傳完整 Runtime 狀態的 JSON。
2. **結構化資料**：使用 `Diagnostics` 結構體定義回傳欄位，確保輸出格式穩定。
3. **檔案匯出**：除了 HTTP 回應，同時寫入 `runtime_diagnostics.json` 檔案供事後分析。
4. **無外部依賴**：不引入 Prometheus、OpenTelemetry 等第三方觀測框架。

### Diagnostics 結構體

定義於 `internal/runtime/diagnostics.go`：

```go
type Diagnostics struct {
    StartupTime     time.Time            `json:"startup_time"`
    ReadyTime       time.Time            `json:"ready_time"`
    ChildStartTime  time.Time            `json:"child_start_time"`
    ShutdownTime    time.Time            `json:"shutdown_time,omitempty"`
    WorkerCount     int                  `json:"worker_count"`
    QueueLength     int                  `json:"queue_length"`
    CurrentState    RuntimeState         `json:"current_state"`
    CurrentStateStr string               `json:"current_state_str"`
    CurrentBackend  string               `json:"current_backend"`
    CurrentModel    string               `json:"current_model"`
    ChildPID        int                  `json:"child_pid"`
    MemoryUsage     int64                `json:"memory_usage_bytes"`
    GoroutineCount  int                  `json:"goroutine_count"`
    ShutdownSteps   []ShutdownStepResult `json:"shutdown_steps,omitempty"`
}
```

### 資料來源

| 欄位 | 來源 |
|------|------|
| `startup_time` | `Runtime.startTime` — `New()` 時記錄 |
| `ready_time` | `Runtime.readyTime` — `StartAudioCpp()` 成功時記錄 |
| `child_start_time` | `Runtime.childStartTime` — child process 啟動時記錄 |
| `shutdown_time` | `Runtime.shutdownTime` — `Shutdown()` 開始時記錄 |
| `worker_count` | `Config.Jobs.Workers` |
| `queue_length` | `JobManager.QueueLen()` |
| `current_state` | `Runtime.currentState()` — 經由狀態機取得 |
| `current_backend` | `Config.AudioCpp.Backend` |
| `current_model` | `Config.AudioCpp.Models[0].ID` |
| `child_pid` | `Process.Pid()` |
| `memory_usage_bytes` | Go runtime 的 `runtime.ReadMemStats().Alloc` |
| `goroutine_count` | `runtime.NumGoroutine()` |
| `shutdown_steps` | `ShutdownSchedule.Steps`（僅 shutdown 後有值） |

### API 端點實作

定義於 `internal/api/handlers_extras.go`：

```go
func (s *Server) handleDiagnostics(w http.ResponseWriter, r *http.Request) {
    d := s.runtimeRef.Diagnostics()
    diagJSON, _ := d.ExportJSON()

    // 同時寫入檔案
    d.ExportToFile(diagPath)

    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    w.Write(diagJSON)
}
```

### 與 Health 的職責劃分

- **`/v1/health`**：服務存活檢查（輕量、高頻呼叫）。返回：status、version、runtime_state、audiocpp_alive、models_count、jobs_pending、uptime。
- **`/v1/runtime/diagnostics`**：詳細內部狀態（重量、低頻呼叫）。返回：完整 Diagnostics 結構體，包含記憶體用量、goroutine 數量、shutdown 歷史等。

### 測試覆蓋

| 測試 | 驗證內容 |
|------|---------|
| `TestDiagnostics_Collect` | 正常狀態下 diagnostics 欄位非空：StartupTime、ReadyTime、CurrentState、ChildPID、GoroutineCount、MemoryUsage |
| `TestDiagnostics_AfterShutdown` | Shutdown 後 diagnostics 的 CurrentState==Stopped、ShutdownTime 非空、JSON export 合法 |

## Alternatives

### Prometheus + Grafana

引入 Prometheus client library，暴露 `/metrics` 端點，配合 Grafana 製作儀表板。

- **優點**：標準業界方案；支援歷史趨勢分析；可設 alert rule。
- **缺點**：引入外部依賴（`prometheus/client_golang`）；需要 Grafana 伺服器；設定複雜；佔用更多系統資源；違反 Phase 6D「不得新增第三方框架」的約束。
- **棄用原因**：Phase 6D 明確禁止引入 Prometheus 等外部觀測框架。對於一個單機 Runtime 來說，diagnostics API 已經足夠。Phase 7 的 Roadmap 中已規劃輕量 Metrics API 作為下一步改進。

### 僅 Logs

不新增 API，僅透過 `log.Printf` 輸出 Runtime 狀態到日誌。

- **優點**：零新增程式碼。
- **缺點**：日誌是文字格式，無法直接機器解析；要取得當前狀態需 grep 大量日誌；無法結構化查詢（如「最近一次 shutdown 的 duration 是多少」）。
- **棄用原因**：不符合可觀測性的基本要求。日誌是除錯工具，不是觀測介面。

### 完整 OpenTelemetry

導入 OpenTelemetry，自動收集 tracing、metrics、logs。

- **優點**：業界標準；未來可與分散式追蹤整合。
- **缺點**：引入重量級依賴；設定複雜；對本專案目前的單一程序架構而言 Overkill。
- **棄用原因**：過度設計。當專案進入 Phase 9（Cluster、Distributed Runtime）時可重新評估。

## Consequences

### 正面影響

1. **足夠的 debug 資訊**：一個 API 呼叫即可取得 Runtime 的完整 snapshot，包含記憶體、goroutine、queue 狀態等。
2. **Shutdown 可追溯**：`shutdown_steps` 欄位記錄了每次 shutdown 的每個步驟的詳細結果（timeout、error、duration），可用於分析 shutdown 失敗原因。
3. **檔案匯出支援**：`runtime_diagnostics.json` 可在 Runtime 停止後供事後分析，不依賴 HTTP API 的可用性。
4. **零外部依賴**：所有資料都來自 Go 標準庫（`runtime.ReadMemStats`、`runtime.NumGoroutine`）和 Runtime 內部欄位，無第三方套件。

### 負面影響

1. **無長期趨勢分析**：diagnostics 是單次快照，沒有歷史資料。無法觀察記憶體增長趨勢、queue 長度變化等時間序列資料。此限制已記錄為 TD-015。
2. **無告警能力**：無法設定閾值告警（如「queue 長度超過 100 時通知」）。需要外部監控系統輪詢 diagnostics API 才能實現。
3. **效能開銷**：每次 diagnostics 呼叫會觸發 Go GC 的 `ReadMemStats`（短暫 STW），以及 `NumGoroutine`（O(1) 但原子操作）。建議呼叫頻率不高於 1 次/秒。

### Trade-off

輕量 diagnostics API 是功能與複雜度的最佳平衡點。在 Phase 6D 的約束下（不引入第三方框架），diagnostics 提供了「足夠好」的可觀測性。長期監控需求已記錄為技術債（TD-015），規劃在 Phase 7 以輕量 Metrics API 解決。
