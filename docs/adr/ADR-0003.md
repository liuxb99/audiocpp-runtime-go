# ADR-0003: Smoke Evidence Model

## 狀態

**Accepted** — 實作於 Phase 6C，提交 `3d31ec0` / `794a3d2` / `f009937`。

## Context

Phase 6 的核心目標是將手工 smoke test 轉變為可復現、可驗證、可移植的真實 ASR 端到端驗收。但在 Phase 6B 之前，smoke evidence 的提交模型存在以下問題：

1. **Commit SHA 語義混淆**：`result.md` 中的 Git Commit 欄位曾經填入「正在運行的目錄的 HEAD」，但若在 smoke 過程中修改了程式碼（如修復 script），則 HEAD 不反映被測試的真正程式碼。
2. **無法信任證據**：一次 smoke 可能基於 dirty working tree，無法保證其他人可以復現相同結果。
3. **證據與源碼混合**：smoke 產出物（`artifacts/smoke/`）與原始碼在同一提交中，審計時無法區分「這是程式碼變更」還是「這是驗收證據」。
4. **缺乏時間戳**：無法確認 evidence 是在 source commit 之後的哪個時間點產生的。

Phase 6C 的需求明確要求修正 Smoke Evidence 與 Commit SHA 的關係。

## Decision

採用 **三欄位證據模型** 與 **Source/Evidence 分離提交**。

### 三欄位模型

`result.md` 中記錄以下三個獨立欄位，取代舊有的單一「Git Commit」欄位：

```
| 欄位 | 說明 |
|------|------|
| **Tested Source Commit** | 執行 smoke 測試前 `git rev-parse HEAD` 的值。此為被測試的程式碼版本。 |
| **Evidence Generated At** | 證據產生的 UTC 時間戳（ISO 8601）。 |
| **Evidence Commit** | 提交 smoke 產物的 commit SHA（若存在）。若無法回寫自身 SHA 則省略，不得寫入錯誤值。 |
```

實作於 `scripts/real_smoke_test.ps1`：

```powershell
$sourceCommit = git rev-parse HEAD
$evidenceTime = (Get-Date).ToUniversalTime().ToString("yyyy-MM-ddTHH:mm:ssZ")
```

### Source/Evidence 分離提交

每次真實驗證分為兩個獨立的 Git commit：

1. **Source Commit**：只包含程式碼、測試、設定檔。提交訊息範例：
   ```
   test: complete runtime integration and backend contracts
   ```
   
2. **Evidence Commit**：只包含 `artifacts/smoke/` 目錄下的 smoke 產物。提交訊息範例：
   ```
   test: record phase 6e runtime integration evidence
   ```

兩次提交的流程：

```
[Source Commit] → 重新編譯 → 執行 real_smoke_test.ps1 → [Evidence Commit]
                    go build -o bin/audiocpp-runtime.exe
```

### 真實範例（Phase 6E）

- Source Commit: `074d4e7` — 程式碼與測試
- Evidence Commit: `af2db6b` — 僅 `artifacts/smoke/`

### PASS/FAIL 判定

最終報告以 `REAL_SMOKE_PASS` 或 `REAL_SMOKE_FAIL` 標記（全大寫、無 BOM、UTF-8），取代舊有的 Emoji 標記。

PASS 條件：
- HTTP 200
- Transcription 內容非空
- 匹配單詞 ≥ 5 且匹配率 ≥ 50%
- Diagnostics HTTP 200 且 JSON 合法
- Runtime 自然退出（External Force Kill Used = False）

## Alternatives

### 單一提交模型

程式碼、測試、smoke 產物全部放在同一個 commit 中。

- **優點**：只需一次 git commit，流程簡單。
- **缺點**：審計時無法區分程式碼與證據；若 smoke 失敗，需 revert 包含產物在內的整個 commit；產物中的時間戳與 commit timestamp 未必一致。
- **棄用原因**：不符合審計鏈的可追溯性要求。Phase 6C 需求明確要求「先提交程式碼修正為 source commit 並推送，再基於 source commit 重新編譯並執行 smoke」。

### 自動回寫 Evidence Commit SHA

smoke script 嘗試在執行過程中將自己的 commit SHA 寫入 `result.md`。

- **缺點**：寫入後需再次 commit 才能包含該 SHA，形成先有雞還是先有蛋的問題。若 smoke 失敗則 dirty working tree 需手動清理。
- **棄用原因**：實務上不可行。`evidence_commit` 欄位的設計就是為了解決此問題——若無法取得則誠實省略，不寫入錯誤值。

## Consequences

### 正面影響

1. **證據可信度高**：`tested_source_commit` 保證了被測試的程式碼版本是可復現的。
2. **審計鏈清晰**：Source Commit 與 Evidence Commit 分離，審計者可獨立驗證程式碼變更與測試證據。
3. **時間戳精確**：`evidence_generated_at` 為 ISO 8601 格式，精確到秒，不受 git commit timestamp 影響。
4. **語義明確**：`REAL_SMOKE_PASS`/`REAL_SMOKE_FAIL` 的二元結果無歧義，且 PASS 條件有量化標準。

### 負面影響

1. **兩次提交增加流程**：每次真實驗證需要兩次 git push，且第二次（evidence commit）需在 smoke 成功後才能執行。
2. **CI 流程複雜度增加**：若未來要在 CI 中自動執行真實驗證，需處理兩次提交的邏輯或使用 CI artifact 替代 git commit。
3. **工作目錄需乾淨**：第一次 Source Commit 後 working tree 必須乾淨，不能有未提交的修改，否則 source SHA 不反映實際測試的程式碼。

### Trade-off

兩次提交的流程成本換來了證據鏈的完整性與可信度。對於 Phase 6 的封板需求——需要可靠的驗收證據以宣告 Phase 6 完成——這是必要的 trade-off。Phase 7 的 CI 自動化可以考慮以 CI artifact 替代 evidence commit，減少流程負擔。
