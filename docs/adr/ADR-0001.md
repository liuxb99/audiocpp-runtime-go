# ADR-0001: Runtime Lifecycle

## 狀態

**Accepted** — 實作於 Phase 6D，提交 `e4dbfb2`。

## Context

專案初期（Phase 1-5）缺乏統一的 Runtime 生命週期管理。`internal/runtime/runtime.go` 中的 `Runtime` 結構體雖然存在，但狀態散落在不同元件中：

- `internal/audiocpp/process.go` 有自己的 `ProcessState`（Stopped、Starting、Running、Stopping、Crashed）。
- API Server 有自己的 `shuttingDown` atomic flag。
- 各個元件各自判斷「是否正在執行」、「是否已停止」，沒有集中式的狀態機。
- 導致 shutdown 順序不可靠：Phase 6C 之前曾發生 worker 已停止但 child process 仍在執行、或 HTTP server 已關閉但資料庫仍被寫入等問題。

Phase 6D 的需求明確要求建立真正的 Runtime 狀態機，將所有狀態遷移集中管理。

## Decision

採用 **七階段狀態機** 作為 Runtime 核心生命週期模型：

```
Created → Initializing → Starting → Ready → Running → Stopping → Stopped
```

七個狀態定義於 `internal/runtime/state.go`：

| 狀態 | 值 | 說明 |
|------|-----|------|
| `StateCreated` | 0 | Runtime 結構體剛建立 |
| `StateInitializing` | 1 | 初始化資料庫、Client、Worker、Process 等元件 |
| `StateStarting` | 2 | 啟動 audio.cpp 子程序 |
| `StateReady` | 3 | audio.cpp 已就緒，可接受任務 |
| `StateRunning` | 4 | Worker Pool 已啟動，正在處理任務 |
| `StateStopping` | 5 | 正在執行關閉程序 |
| `StateStopped` | 6 | 完全停止 |

實作細節：

1. **狀態遷移表** 使用 `validTransitions` map 定義合法路徑：

   ```
   Created      → {Initializing, Stopping}
   Initializing → {Starting, Stopping}
   Starting     → {Ready, Stopping}
   Ready        → {Running, Stopping}
   Running      → {Stopping}
   Stopping     → {Stopped}
   Stopped      → {}
   ```

2. **雙層鎖保護**：`stateMu`（sync.Mutex）保護所有狀態讀寫；`transition()` 和 `transitionLocked()` 兩個方法執行遷移檢查。

3. **Shutdown 全局准入**：`transitionFromAnyToStoppingLocked()` 允許從任何活躍狀態直接轉入 Stopping，確保 shutdown 請求永遠不會被拒絕。

4. **Process 層級也有自己的狀態機**（`ProcessState`），但 `Runtime` 狀態機是最高權威，所有外部查詢統一透過 `Runtime.CurrentState()`。

## Alternatives

### 無狀態模型

每一請求獨立啟動/關閉 audio.cpp，不維護 Runtime 狀態。

- **缺點**：每次請求需等待模型載入（大型模型 > 30 秒），完全無法接受；無法支援 job queue 背景處理；無 worker pool 重用。
- **棄用原因**：效能不可接受，且違反「長期執行服務」的架構前提。

### 簡易兩狀態模型（Running / Stopped）

- **缺點**：無法表達初始化中、啟動中、準備就緒等中間狀態；shutdown 過程無細粒度控制，無法區分 graceful stop 與 force kill。
- **棄用原因**：Phase 6C 的 shutdown 語義修正需求直接證明了兩狀態模型不足——無法區分「請求被接受」、「優雅退出完成」、「強制終止已使用」三種語義。

### Event-driven 架構

以事件匯流排（Event Bus）驅動狀態變更，各元件訂閱事件。

- **缺點**：引入非同步複雜性，狀態變更時序難以追蹤；Phase 6D 的需求強調「單一路徑、禁止跳躍」，事件驅動會讓遷移路徑變得隱式。
- **棄用原因**：複雜度與可控性權衡不符本專案現階段需求。

## Consequences

### 正面影響

1. **明確的狀態語義**：任何時刻可透過 `CurrentState()` 取得 Runtime 的準確狀態，不再依賴元件各自判斷。
2. **狀態遷移可驗證**：每個轉移都有 Contract Test 覆蓋（`TestRuntimeLifecycle_FullCycle`、`TestStateTransition_DoubleShutdown`、`TestStateTransition_InvalidTransition` 等）。
3. **Shutdown 安全**：從任何狀態都可經由 `transitionFromAnyToStoppingLocked()` 進入關閉流程，不會卡在非法狀態。
4. **可觀測性**：狀態直接暴露在 `/v1/health` 和 `/v1/runtime/diagnostics` 中，運維人員可即時查看。

### 負面影響

1. **狀態檢查增加開銷**：每次 `transition()` 呼叫需取得 mutex lock、查表、驗證，但此開銷在微秒級，對整體效能影響可忽略。
2. **新增狀態需修改遷移表**：若要新增狀態（如 Paused、Degraded），需同時修改 `validTransitions`、`StateString`、以及所有呼叫 `transition()` 的位置。
3. **Process 層級狀態機重疊**：`Runtime` 與 `Process` 各有自己的狀態機，兩者可能暫時不一致（例如 Process 已 crashed 但 Runtime 仍在 Running）。需透過 `supervise()` 監控機制最終收斂。

### Trade-off

狀態機強度換來了可預測性與可測試性。對於一個需要管理外部子程序生命週期的系統，這種 trade-off 是合理的——狀態機的嚴謹性是 Runtime 穩定性的基石。
